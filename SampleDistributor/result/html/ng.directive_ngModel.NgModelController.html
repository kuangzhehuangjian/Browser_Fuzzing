Description

code provides API for the code directive. The controller contains services for data-binding, validation, CSS updates, and value formatting and parsing. It purposefully does not contain any logic which deals with DOM rendering or listening to DOM events. Such DOM related logic should be provided by other directives which make use of code for data-binding.

Custom Control Example

This example shows how to use code with a custom control to achieve data-binding. Notice how different directives (code, code, and code) collaborate together to achieve the desired result.

Note that code is an HTML5 attribute, which tells the browser to let the element contents be edited in place by the user. This will not work on older browsers.

Methods

  • $render()

    Called when the view needs to be updated. It is expected that the user of the ng-model directive will implement this method.

  • $setPristine()

    Sets the control to its pristine state.

    This method can be called to remove the 'ng-dirty' class and set the control to its pristine state (ng-pristine class).

  • $setValidity(validationErrorKey, isValid)

    Change the validity state, and notifies the form when the control changes validity. (i.e. it does not notify form if given validator is already marked as invalid).

    This method should be called by validators - i.e. the parser or formatter functions.

    Parameters
    ParamTypeDetails
    validationErrorKeystring

    Name of the validator. the code will assign to code so that it is available for data-binding. The code should be in camelCase and will get converted into dash-case for class name. Example: code will result in code and code class and can be bound to as code .

    isValidboolean

    Whether the current state is valid (true) or invalid (false).

  • $setViewValue(value)

    Update the view value.

    This method should be called when the view value changes, typically from within a DOM event handler. For example code and code directives call it.

    It will update the $viewValue, then pass this value through each of the functions in code, which includes any validators. The value that comes out of this code pipeline, be applied to code and the expression specified in the code attribute.

    Lastly, all the registered change listeners, in the code list, are called.

    Note that calling this function does not trigger a code.

    Parameters
    ParamTypeDetails
    valuestring

    Value from the view.

  • $isEmpty(value)

    This is called when we need to determine if the value of the input is empty.

    For instance, the required directive does this to work out if the input has data or not. The default code function checks whether the value is code, code, code or code.

    You can override this for input directives whose concept of being empty is different to the default. The code directive does this because in its case a value of code implies empty.

    Parameters
    ParamTypeDetails
    value*

    Reference to check.

    Returns
    boolean

    True if code is empty.

Properties

  • $viewValue

    Actual string value in the view.

  • $modelValue

    The value in the model, that the control is bound to.

  • $parsers

    Array of functions to execute, as a pipeline, whenever the control reads value from the DOM. Each function is called, in turn, passing the value through to the next. Used to sanitize / convert the value as well as validation. For validation, the parsers should update the validity state using code, and return code for invalid values.

  • $formatters

    Array of functions to execute, as a pipeline, whenever the model value changes. Each function is called, in turn, passing the value through to the next. Used to format / convert values for display in the control and validation. 

    pre

  • $viewChangeListeners

    Array of functions to execute whenever the view value has changed. It is called with no arguments, and its return value is ignored. This can be used in place of additional $watches against the model value.

  • $error

    An object hash with all errors as keys.

  • $pristine

    True if user has not interacted with the control yet.

  • $dirty

    True if user has already interacted with the control.

  • $valid

    True if there is no error.

  • $invalid

    True if at least one error on the control.
