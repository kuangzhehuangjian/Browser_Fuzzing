    Long Version

    IE has issues with element tag names which are not standard HTML tag names. These fall into two categories, and each category has its own fix.

    • If the tag name starts with code prefix then it is considered an XML namespace and must have corresponding namespace declaration on code

    • If the tag has no code but it is not a standard HTML tag, then it must be pre-created using code

    • If you are planning on styling the custom tag with CSS selectors, then it must be pre-created using code regardless of XML namespace.

    The Good News

    The good news is that these restrictions only apply to element tag names, and not to element attribute names. So this requires no special handling in IE: code.

    What happens if I fail to do this?

    Suppose you have HTML with unknown tag code (this could also be code or code with same result):

    pre

    It should parse into the following DOM:

    pre

    The expected behavior is that the code element has a child element code, which in turn has the text code.

    But this is not what IE does (if the above fixes are not included):

    pre

    In IE, the behavior is that the code element has three children:

    1. A self closing code. Example of self closing tag is code. The trailing code is optional, but the code tag is not allowed to have any children, and browsers consider code as three siblings not a code with code as child.

    2. A text node with code. This should have been a child of code above, not a sibling.

    3. A corrupt self closing code. This is corrupt since element names are not allowed to have the code character. Furthermore this closing element should not be part of the DOM since it is only used to delineate the structure of the DOM.
